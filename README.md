# Southwest Toy Airline Data

A local data stack for querying Southwest Airlines public schedule, aircraft rotation, and fare data via an LLM. Built as a proof of concept.

**Stack:** XLSX ‚Üí Delta Lake ‚Üí Delta Sharing (REST) ‚Üí MCP Server ‚Üí LLM

---

## Prerequisites

- Python 3.12+
- The raw XLSX data files placed in `raw_data/` (not included in this repo)

---

## Setup

```zsh
# Clone the repo
git clone <repo-url>
cd toy-airline-data

# Add your XLSX files to raw_data/
# Expected files:
#   1_schedules_OAG_May.xlsx
#   1_schedules_OAG_Nov.xlsx
#   1a_schedules_WN_departures_and_arrivals.xlsx
#   1b_schedules_market_and_competition.xlsx
#   2_aircraft_rotation_FR24_Apr.xlsx
#   2a_aircraft_rotation_analysis.xlsx
#   3_fares_and_fare_analysis.xlsx

# Bootstrap the environment (creates venv/, installs deps, sets up DuckDB)
make setup
```

## Ingest Data

```zsh
# Preview what will be ingested (no writes)
make dry-run

# Convert all XLSX files ‚Üí Delta tables in delta_tables/
make ingest
```

This produces 18 Delta tables in `delta_tables/`. The following files/sheets are intentionally excluded:

| Excluded | Reason |
|---|---|
| `1c_WN_lounge_analysis.xlsx` | Interactive Excel simulation model, not raw data |
| All `S` sheets | Sources/legend tabs (22-32 rows each) |

## Adapting to Different Data

The ingestion pipeline is mostly generic ‚Äî it will process any XLSX files placed in `raw_data/`. However, two things are specific to this dataset and will need updating if you use different files:

### 1. Exclusion lists (`ingestion/ingest.py` and `ingestion/excel_reader.py`)

```python
# ingestion/ingest.py ‚Äî skip entire files
SKIP_FILES = {"1c_WN_lounge_analysis.xlsx"}

# ingestion/excel_reader.py ‚Äî skip specific sheet names (by slug)
SKIP_SHEETS = {"s"}
```

- `SKIP_FILES` ‚Äî add filenames that are dashboard/simulation models rather than data tables
- `SKIP_SHEETS` ‚Äî add sheet name slugs that are metadata tabs (sources, legends, etc.)

Run `make dry-run` first to see exactly what sheets each file contains before committing to a full ingest.

### 2. `sharing/sharing_config.yaml`

This file is **fully manual** ‚Äî it maps the Delta Sharing protocol hierarchy (share ‚Üí schema ‚Üí table) to the Delta table paths generated by ingestion. You must update it whenever you change your data files.

**Table names are generated by ingestion** using this convention:
- Single-sheet file: `{file_stem}` (slugified)
- Multi-sheet file: `{file_stem}__{sheet_slug}`

After running `make ingest`, check `ls delta_tables/` to see the exact names, then update `sharing_config.yaml` to match:

```yaml
shares:
  - name: my_share
    schemas:
      - name: my_schema
        tables:
          - name: friendly_name       # what the MCP tools and LLM will see
            location: delta_tables/actual_delta_table_directory_name
```

The `name` field is what the LLM queries against ‚Äî choose something readable. The `location` must match the exact directory name under `delta_tables/`.

## Running the Servers

Two servers must run simultaneously. Open two terminal tabs:

**Terminal 1 ‚Äî Delta Sharing Server** (port 8080)
```zsh
make sharing-server
```

**Terminal 2 ‚Äî MCP Server** (stdio)
```zsh
make mcp-server
```

The MCP server reads all data through the sharing server ‚Äî it does not access Delta tables directly.

## Connecting to Claude Desktop

1. Open or create `~/Library/Application Support/Claude/claude_desktop_config.json`

2. Add the following entry (adjust the path to match your clone location):

```json
{
  "mcpServers": {
    "Toy Airline": {
      "command": "/your/path/to/toy-airline-data/venv/bin/python",
      "args": ["-m", "mcp_server.server"],
      "cwd": "/your/path/to/toy-airline-data",
      "env": {
        "PYTHONPATH": "/your/path/to/toy-airline-data"
      }
    }
  }
}
```

3. Start the sharing server: `make sharing-server`

4. Restart Claude Desktop ‚Äî the MCP server launches automatically as a subprocess.

5. In a new Claude chat, the following tools will be available:
   - `list_tables` ‚Äî see all available tables
   - `get_schema` ‚Äî column names and types for a table
   - `sample_data` ‚Äî preview rows from a table
   - `query_table` ‚Äî run SQL against the data

**Example prompts:**
- *"What tables are available?"*
- *"Show me the schema for the oag_may table"*
- *"What are the top 10 origin airports by flight count in May?"*
- *"Compare seat capacity between May and November for LAS"*

## Connecting to Other LLM Clients

The MCP server supports two transports:

### stdio (default)
Used by Claude Desktop and any MCP-compatible client that launches subprocesses. No extra config needed ‚Äî the client manages the process lifecycle.

### SSE / HTTP
For clients that connect over HTTP (e.g. custom apps, other LLM frameworks):

```zsh
make mcp-server-sse   # starts on port 8081
```

Then point your client at `http://localhost:8081/sse`.

### Using with the Anthropic API directly

```python
import anthropic

client = anthropic.Anthropic()

response = client.beta.messages.create(
    model="claude-opus-4-6",
    max_tokens=1024,
    mcp_servers=[{
        "type": "url",
        "url": "http://localhost:8081/sse",
        "name": "toy-airline"
    }],
    messages=[{"role": "user", "content": "What tables are available?"}],
    betas=["mcp-client-2025-04-04"],
)
print(response.content)
```

### Using with other MCP-compatible frameworks

Any framework that implements the [Model Context Protocol](https://modelcontextprotocol.io) can connect to this server. The four tools exposed are:

| Tool | Input | Description |
|---|---|---|
| `list_tables` | _(none)_ | List all available tables |
| `get_schema` | `table_name: str` | Column names and types |
| `sample_data` | `table_name: str`, `n: int` (default 10) | First N rows |
| `query_table` | `sql: str` | SQL query (DuckDB dialect, 100-row default cap) |

SQL queries reference tables by short name ‚Äî the server resolves them automatically:
```sql
SELECT origin, COUNT(*) as flights
FROM oag_may
GROUP BY origin
ORDER BY flights DESC
LIMIT 20
```
